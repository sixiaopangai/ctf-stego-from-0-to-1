# README 写作规范

## 1. 适用范围

这份规范适用于：

- 仓库顶层 `README.md`
- `practice/README.md`
- `scripts/README.md`
- 后续新增子目录的说明页

目标是让所有 README 都具备统一的结构、清晰的导航能力和较低的维护成本。

## 2. 核心原则

- 一个 README 只解决一个层级的问题
- 先说“这是什么”，再说“怎么用”，最后说“去哪里继续看”
- README 负责导航，不负责承载全部正文
- 任何链接都优先使用相对路径
- 所有命令示例都要可直接复制执行
- 能在子目录说明清楚的内容，不要堆回顶层首页

## 3. 顶层 README 推荐结构

顶层 `README.md` 建议保持以下顺序：

1. 项目标题
2. 项目简介
3. 项目定位 / 适合人群
4. 仓库结构总览
5. 推荐阅读或使用路径
6. 重要入口链接
7. 文档维护约定
8. 发布或协作提醒

其中：

- “项目简介”建议控制在 2 到 4 句话
- “仓库结构”优先用表格或短列表
- “推荐路径”要能直接帮助第一次访问者做选择

## 4. 子目录 README 推荐结构

子目录 README 建议保持以下顺序：

1. 目录定位
2. 目录内容概览
3. 使用方式
4. 注意事项
5. 指向上级或相关文档的链接

例如：

- `practice/README.md` 更关注做题顺序、剧透边界、目录结构
- `scripts/README.md` 更关注依赖、用途和命令示例

## 5. 标题与层级规则

- 每个 README 只能有一个 `#` 一级标题
- 常规情况下标题层级不超过 `###`
- 同一级标题的命名风格保持一致
- 不要为了视觉效果跳级，例如从 `#` 直接跳到 `###`

## 6. 内容写法规则

### 6.1 简介段

- 第一段必须明确项目对象、主题和价值
- 避免只写口号，不写具体内容
- 避免上来就是超长背景介绍

### 6.2 列表与表格

- 导航类信息优先使用列表或表格
- 列表项尽量同时包含“路径 + 一句话说明”
- 当信息需要横向比较时，用表格优于长段落

### 6.3 链接

- 优先使用相对路径链接
- 链接文本应体现目标内容，不建议只写“点击这里”
- 首页必须能跳到练习区、脚本区和规范文档

### 6.4 命令示例

- 代码块必须标注语言，例如 `powershell`
- 命令示例默认以仓库根目录为执行位置
- 示例命令尽量完整，不省略关键参数

### 6.5 信息边界

- README 中不要直接泄露练习答案
- `practice/writeups-index.md` 这类带剧透内容必须显式提示
- 大段正文知识应放回章节文档，而不是复制到 README

## 7. 维护要求

当以下内容变化时，必须检查 README 是否需要同步更新：

- 目录结构变化
- 文件名变化
- 练习题数量变化
- 脚本依赖或命令变化
- 对外展示定位变化

维护时优先保证：

1. 链接不失效
2. 首页路径清晰
3. 内容不重复
4. 说明与实际目录一致

## 8. 不推荐的写法

- 把完整教程正文直接塞进首页
- 首页出现大量答案或剧透
- 只列文件名，不解释用途
- 命令示例缺少执行前提
- 一个 README 同时承担导航、正文、FAQ、题解四种职责

## 9. 建议的检查清单

提交 README 相关改动前，至少检查以下项目：

1. 是否只有一个一级标题
2. 首页是否能说明项目是什么
3. 关键目录是否都有可点击入口
4. 子目录 README 是否承担了各自职责
5. 是否误放了答案、临时说明或本地路径
