# 项目文档规划

## 1. 规划目标

这份仓库面向 GitHub 展示时，文档需要同时满足三件事：

1. 让第一次进入仓库的人能在 1 分钟内看懂项目做什么
2. 让学习者能快速找到“章节讲义、练习题、脚本、完整版导出”各自入口
3. 让后续维护者知道什么内容该放首页，什么内容该下沉到子目录文档

当前仓库内容已经比较完整，因此本轮规划不做大规模移动，只通过“首页重构 + 子目录说明页 + 规范文档”来完成重组。

## 2. 文档分层

| 层级 | 位置 | 作用 | 维护原则 |
| --- | --- | --- | --- |
| L0 | [`README.md`](../README.md) | 仓库首页与总导航 | 只回答“这是什么、从哪里开始、去哪里看” |
| L1 | 根目录 `01-08` 章节文档 | 教程主内容 | 专注知识讲解，不写仓库管理信息 |
| L1 | [`practice/README.md`](../practice/README.md) | 练习区总入口 | 说明题目顺序、使用方法、剧透边界 |
| L1 | [`scripts/README.md`](../scripts/README.md) | 脚本入口 | 说明依赖、用途、命令示例 |
| L1 | [`docs/README写作规范.md`](./README写作规范.md) | README 规范 | 统一结构、风格与更新要求 |
| L2 | `practice/*/challenge.md` / `hint.md` / `solution.md` | 单题说明 | 单题内聚，不跨题引用无关内容 |
| L2 | `practice/writeups-index.md` | 答案汇总 | 明确剧透提示，不在首页展开 |
| L3 | `CTF隐写从0到1-完整版.*` | 导出版讲义 | 作为整合阅读版本，不替代首页导航 |

## 3. 各模块职责

### 3.1 顶层 README

顶层 `README.md` 必须承担仓库首页职责，建议固定回答以下问题：

- 这是什么项目
- 适合谁
- 仓库里有哪些内容
- 建议从哪里开始
- 实操练习和脚本入口在哪里
- 项目规范放在哪里

不建议在顶层 `README.md` 中放入以下内容：

- 全量教程正文
- 详细单题解法
- 大段重复复制自章节文档的内容
- 难以维护的超长命令清单

### 3.2 教程章节

根目录 `01-08` 的编号章节继续保持当前形态，用来承载主学习内容。后续新增章节时，仍建议保持 `NN-主题名.md` 的命名方式，便于排序与导航。

### 3.3 练习区

`practice/` 负责“做题”而不是“讲概念”，建议维持以下边界：

- `README.md`：总入口、顺序、使用方式、批次划分
- `challenge.md`：只给题面
- `hint.md`：只给提示
- `solution.md`：只给参考解法
- `writeups-index.md`：统一答案汇总，但必须标注含剧透

### 3.4 脚本区

`scripts/` 负责配套工具说明。脚本 README 应明确：

- 适用场景
- 依赖条件
- 可直接复制执行的命令示例
- 是否会生成或覆盖本地文件

### 3.5 docs 目录

`docs/` 只放“项目级元文档”，不放教程正文。它解决的是维护与协作问题，例如：

- 仓库结构规划
- README 写作规范
- 发布检查清单
- 贡献约定

## 4. 命名与组织约定

- 教程章节：`NN-主题名.md`
- 练习目录：`NN-topic-keywords/`
- 单题附件：统一放在 `files/`
- 子目录入口页：统一命名为 `README.md`
- 项目级规范文档：放在 `docs/`

## 5. 更新规则

出现以下变更时，应同步更新顶层 `README.md`：

- 新增或删除章节
- 新增或删除练习题
- 新增脚本或脚本依赖变化
- 整合版导出文件名称变化
- 仓库结构调整

出现以下变更时，应同步更新对应子目录 `README.md`：

- `practice/` 的题目数量、顺序、批次变化
- `scripts/` 的命令示例或依赖变化

只有当维护规则本身改变时，才需要更新 `docs/` 下的规范文档。

## 6. GitHub 发布建议

当前仓库已经适合直接发布到 GitHub，但建议遵循以下检查项：

1. 确认 `.gitignore` 已过滤 Word 临时文件与 `node_modules`
2. 确认首页、练习区、脚本区 README 的相对链接都可打开
3. 确认 `practice/writeups-index.md` 明确标注剧透
4. 确认大文件体积仍在常规 Git 管理范围内
5. 若后续加入更多媒体资源，再评估是否切换 Git LFS

## 7. 后续可选优化

如果仓库继续扩展，可以再考虑以下动作，但当前不是必须：

- 增加 `LICENSE`
- 增加 `CONTRIBUTING.md`
- 为章节建立更细的目录索引页
- 为练习区补充按题型分类索引
